# ADR-0005

## Title

Database-generated IDs and timestamps over application-generated values.

## Status

Accepted

## Context

Every row needs a primary key and audit timestamps. These values can be
generated in two places: the application (in Go, before the INSERT) or
the database (via Postgres DEFAULT expressions on the column).

The question is which layer is the correct owner of this responsibility.

## Decision

Let Postgres generate all primary keys and timestamps.

Primary keys use DEFAULT gen_random_uuid() on the column definition.
created_at uses DEFAULT NOW(). updated_at is set to NOW() explicitly
in each UPDATE statement.

The application never generates IDs or timestamps. It uses RETURNING
to read the generated values back after each INSERT or UPDATE.

## Consequences

Advantages

- Atomicity. An ID only exists if the row was committed. If an INSERT
  fails, no ID was produced. Application-generated IDs can exist and
  be referenced before the corresponding row is confirmed in the
  database.
- updated_at uses the database clock, which is a single source of
  truth. Application servers in a multi-instance deployment can have
  clock drift between them. The database clock does not.
- RETURNING retrieves the generated values in the same round-trip as
  the write. No second SELECT query is needed after an INSERT.
- UUIDs generated by Postgres are not predictable or sequential,
  which avoids exposing record counts through the API.

Disadvantages

- The application does not know the ID of a record before inserting it,
  which means it cannot construct relationships or references ahead of
  the write.
- Requires RETURNING in every INSERT, which is Postgres-specific and
  would not port to databases that do not support it.
